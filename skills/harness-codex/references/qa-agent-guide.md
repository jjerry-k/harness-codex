# QA Agent Guide

메타 하네스를 설계할 때 QA 역할을 어떻게 넣을지 판단하기 위한 가이드입니다.

## QA 역할이 필요한 신호
아래 중 하나라도 강하면 QA 역할을 별도로 두는 편이 좋습니다.

- 프론트와 백엔드가 동시에 바뀜
- API 계약이 자주 깨질 위험이 있음
- 상태 전이가 복잡함
- 라우팅/권한/폼 검증이 얽혀 있음
- 기존 운영 서비스라 회귀 리스크가 큼
- speed보다 stability/quality가 더 중요함

## QA가 놓치면 안 되는 결함 패턴

### 1. Boundary mismatch
서로 맞물리는 두 영역이 어긋나는 경우입니다.

예:
- API 응답 필드명과 프론트 사용 필드명이 다름
- 라우터 경로와 링크 경로가 다름
- 상태 전이 정의와 실제 UI 동작이 다름
- 프론트가 호출하는 엔드포인트와 서버 구현이 다름

### 2. Static review로 잘 안 잡히는 문제
단일 파일 리뷰만으로는 아래가 잘 안 보입니다.

- 프론트 훅 ↔ API 응답 구조 불일치
- 버튼 액션 ↔ 실제 페이지/엔드포인트 연결 누락
- 정상 흐름은 되지만 error/empty/loading가 깨지는 경우

## QA 설계 원칙

### 1. 존재 확인보다 교차 비교 우선
QA는 "테스트가 있다"보다 아래를 먼저 봐야 합니다.

- 양쪽이 같은 계약을 공유하는가
- 화면/서버/라우팅/상태가 서로 일치하는가

### 2. 양쪽을 동시에 읽게 설계
좋은 QA 역할은 한쪽만 보지 않습니다.

예:
- API 스펙과 프론트 소비 코드 같이 보기
- 라우터 정의와 내비게이션 링크 같이 보기
- 상태 전이 다이어그램과 실제 버튼 흐름 같이 보기

### 3. 마지막 한 번만 하지 말 것
QA는 릴리즈 직전 1회성으로만 두기보다,
큰 모듈이 끝날 때마다 점검하도록 설계하는 편이 좋습니다.

## QA 역할을 넣을지 판단하는 빠른 기준

### QA를 별도 역할로 둘 것
- medium/high risk
- 운영 서비스 변경
- 다중 컴포넌트 통합
- reviewer만으로 부족한 경우

### reviewer에 흡수해도 되는 경우
- low risk
- 아주 작은 범위 변경
- 단일 파일/단일 계층 수정

## QA 출력 형식 권장
- test scope
- passed checks
- failed checks
- reproduction steps
- release blockers
- can defer later
